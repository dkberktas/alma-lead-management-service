# Architecture Snapshot — 2026-03-04

Final specification of the Alma Lead Management Service as delivered.

---

## System Overview

A FastAPI backend and thin React admin client for managing immigration-law prospect intake. Prospective clients submit their information and resume through a public form; attorneys and admins triage leads through a protected internal interface.

---

## API Surface

### Public (no authentication)

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/api/leads` | Submit a new lead (multipart form with resume) |
| POST | `/api/auth/login` | Obtain a JWT access token |
| GET | `/health` | Liveness check (DB + audit subsystem) |

### Attorney / Admin (JWT Bearer)

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/leads` | List leads with state filter and pagination |
| GET | `/api/leads/{id}` | Retrieve a single lead |
| PATCH | `/api/leads/{id}` | Transition lead state (PENDING → REACHED_OUT) |
| GET | `/api/leads/{id}/resume-url` | Presigned download URL for the resume file |

### Admin Only

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/api/admin/attorneys` | Create a new attorney account |
| GET | `/api/admin/users` | List all users |
| GET | `/api/admin/users/{id}` | Get user details |
| PATCH | `/api/admin/users/{id}/deactivate` | Soft-deactivate a user |
| PATCH | `/api/admin/users/{id}/reactivate` | Reactivate a user |
| DELETE | `/api/admin/users/{id}` | Hard-delete a user |
| GET | `/api/admin/files` | List uploaded resume files |
| GET | `/api/admin/audit-logs` | Query audit trail (filterable, paginated) |
| GET | `/api/leads/{id}/audit-log` | Per-lead audit history |

## Data Model

### Lead

UUID primary key. Fields: first_name, last_name, email, resume_path, state (PENDING | REACHED_OUT), created_at, updated_at, reached_out_at, reached_out_by_id (FK → users).

### User

UUID primary key. Fields: email (unique), hashed_password (bcrypt), role (ADMIN | ATTORNEY), is_active, created_at.

### AuditLog

UUID primary key. Fields: entity_type, entity_id, action, user_id, user_email (denormalized), old_state, new_state, detail, lead_id, created_at. Indexed on entity_type, entity_id, action, and created_at.

---

## Authentication & Authorization

- **JWT (HS256)** — issued at login, 60-minute expiry, payload carries user ID and role.
- **Password hashing** — bcrypt via passlib; plaintext is never stored.
- **Role-based access** — two roles: `ADMIN` and `ATTORNEY`. FastAPI `Depends()` guards enforce role requirements per endpoint.
- **Adding new attorneys is an admin operation** — the admin creates accounts through `/api/admin/attorneys`. There is no self-registration. More attorneys can be onboarded at any time without code changes.

---

## Audit Trail

Every significant action is recorded in the `audit_logs` table:

- Lead created, state changed, resume downloaded
- User created, deactivated, reactivated, deleted

Each entry captures who performed the action, the before/after state, and a timestamp. The audit service runs in the background with its own database session so a logging failure never blocks the request path. A built-in failure tracker escalates to CRITICAL log level after three consecutive write failures and self-heals when writes resume.

Admins can query the full trail via `/api/admin/audit-logs` (filterable by entity type and action, paginated) or view per-lead history via `/api/leads/{id}/audit-log`.

## Notification System — Pluggable Channels

When a new lead is submitted the system sends:

1. A confirmation email to the prospect.
2. A notification email to all active attorneys.

The notification layer uses a **channel abstraction** (`services/channels/`). Three channels ship today:

| Channel | When Used |
|---------|-----------|
| `ResendEmailChannel` | Resend API key is configured (preferred) |
| `SmtpEmailChannel` | SMTP credentials are configured (Gmail, etc.) |
| `LogChannel` | Fallback — logs to console when no email provider is set |

**New channels can be added** by implementing a single `send()` method and registering the channel — no changes to the notification service or routes are required.

---

## File Storage — Pluggable Backends

Resume uploads are validated by magic-byte inspection (PDF and DOCX only) and stored under UUID-based keys to prevent filename collisions.

| Backend | Config | Notes |
|---------|--------|-------|
| `LocalStorageBackend` | `STORAGE_BACKEND=local` | Writes to `uploads/` on disk |
| `S3StorageBackend` | `STORAGE_BACKEND=s3` | Uses boto3; presigned URLs (5 min TTL) |

Switching backends is a configuration change — no code modifications needed.

---

## Rate Limiting

Public lead submission is rate-limited per client IP:

- **5 requests per minute**
- **20 requests per hour**

Exceeded limits return `429 Too Many Requests` with a `Retry-After` header.

## Docker Architecture

The production deployment is fully containerized via Docker Compose:

```
┌──────────────────────────────────────────────┐
│  docker-compose                              │
│                                              │
│  ┌────────────┐       ┌──────────────────┐   │
│  │  postgres   │◄──────│     app          │   │
│  │  16-alpine  │       │  python:3.12     │   │
│  │             │       │                  │   │
│  │  pgdata vol │       │  alembic migrate │   │
│  │  init-db.sql│       │  uvicorn :8000   │   │
│  └────────────┘       └──────────────────┘   │
│                              │               │
│                         healthcheck          │
│                     curl /health             │
└──────────────────────────────────────────────┘
```

- **db** — PostgreSQL 16 (Alpine) with a persistent volume and an init script that creates the `alma_app` database role with least-privilege grants.
- **app** — Python 3.12 slim image. On startup it runs Alembic migrations, then launches Uvicorn. A Docker healthcheck polls `/health` to verify both the database connection and the audit subsystem.
- **RLS** — Row-Level Security policies on `leads`, `users`, and `audit_logs` provide defense-in-depth alongside application-layer auth (see ADR-004).

## Thin Web Client (React Admin UI)

A lightweight React SPA lives in `frontend/admin/` and serves as the internal attorney/admin interface.

**Stack:** React 18, Vite 6, TypeScript, Tailwind CSS, React Router 6.

**Pages:**

| Page | Role | Features |
|------|------|----------|
| Login | Public | Email/password form, JWT stored in localStorage |
| Leads | Attorney+ | List, filter by state (ALL / PENDING / REACHED_OUT), paginate, mark as reached out |
| Attorneys | Admin | Create new attorney accounts, deactivate/reactivate existing ones |
| Audit Trail | Admin | Browse full audit log with entity-type and action filters, pagination |

The client is intentionally thin — all business logic lives in the backend. The UI communicates exclusively through the API documented above. Environment targeting (local, staging, production) is handled via a `?api=` query parameter.

## Extensibility Summary

| Dimension | How to Extend |
|-----------|---------------|
| **Notification channels** | Implement `send()` in a new channel class under `services/channels/` — e.g., Slack, SMS, webhook |
| **Storage backends** | Add a new backend class alongside Local and S3 — e.g., GCS, Azure Blob |
| **User roles** | The role enum and `Depends()` guards are the only touchpoints |
| **New attorneys** | Admin creates accounts via API — no deployment or code change needed |
| **Lead states** | Extend the `LeadState` enum and update transition logic in the service layer |
